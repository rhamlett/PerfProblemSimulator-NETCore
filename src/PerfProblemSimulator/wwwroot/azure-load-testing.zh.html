📈 Azure 负载测试指南

为 Azure 负载测试集成设计的负载测试端点

🔬 概述

性能问题模拟器包含专用的负载测试终端 设计用于 Azure 载荷测试。该端点模拟真实感 应用行为在负载下会优雅地下降。

该 /api/loadtest 端点设计目的是：

  • 执行持续 CPU 工作（交错的SHA256周期，每线程 CPU 约50%）
  • 在整个请求期间保持内存缓冲区（产生可见的内存压力）
  • 随着并发请求增加（软限制模式），性能下降
  • 在可配置阈值（默认：120秒，概率20%）后抛出随机异常。
  • 最终在极高负载下触发 Azure230秒的超时

⚙️ 设置 Azure 负载测试

  1. 在 Azure 门户中，创建一个 Azure 负载测试资源
  2. 创建一个新测试，并使用以下选项之一配置目标URL：

GET 查询参数

直接使用目标网址——无需 JMeter 脚本：

# 默认设置（针对Premium0V3调校）https://your-app.azurewebsites.net/api/loadtest# 自定义参数https://your-app.azurewebsites.net/api/loadtest?workIterations=500&bufferSizeKb=10000# 仅线池（无 CPU 工作）https://your-app.azurewebsites.net/api/loadtest?workIterations=0&bufferSizeKb=100
  1. 配置负载模式（增加、稳态、下降）
  2. 设置 Azure 监控集成以关联指标

📋 查询参数参考

每个参数独立控制特定资源：

参数 默认 目标资源 描述
workIterations 200 CPU 每个 CPU 工作周期的 SHA256 哈希迭代。控制持续 CPU 使用率（每线程约50%）。
bufferSizeKb 20000 记忆 内存以KB为单位，整个请求持续时间内保持（默认为20 MB）。会造成持续的记忆压力。
baselineDelayMs 500 线程池 最小请求时长（毫秒）。 CPU 工作和睡眠交替进行。
softLimit 25 线程池 降级前并发请求。低 = 更快的疲劳。
degradationFactor 500 线程池 每个请求在softLimit上额外延迟（毫秒）。
errorAfter 120 混沌 在出现随机错误前几秒钟。设置为0以禁用。
errorPercent 20 混沌 阈值后每个检查间隔随机出现错误的概率（0-100）。

📊 理解软极限

软极限形成一个退化曲线，响应时间随 并发请求超过阈值。降解公式为：

additionalDelay = max（0， concurrent - softLimit） × degradationFactor

注： 实际响应时间会根据你当前的 Azure App Service 等级而异 负荷，以及其他因素。利用 Azure 负载测试来建立基线性能 针对你具体的环境。默认参数是为 Premium0V3 工作者调校的。 其他的 SKU 或工种尺寸都需要单独调校。

降解模式示例

默认设置下 （softLimit=25, degradationFactor=500):

并行请求 额外延迟 预期响应时间
10 0毫秒 ~500毫秒（仅基线）
25 0毫秒 ~500毫秒（软限制）
50 12,500毫秒 ~13
100 37,500毫秒 ~38

💥 错误注入

负载测试端点可以注入随机异常以模拟真实情况 应用程序失败。这由两个参数控制：

  • errorAfter （默认：120）- 错误开始前几秒
  • errorPercent （默认：20）- 达到阈值后出现错误的概率

例外类型

从一系列真实的 .NET 例外中随机选择：

例外类型 示例消息
InvalidOperationException 由于当前状态，操作无效
ArgumentOutOfRangeException 价值不在预期区间内
NullReferenceException 对象引用未设置为对象的实例
KeyNotFoundException 给出的密钥未出现在词典中
TimeoutException 行动已超时
IOException 无法读取传输连接的数据
OutOfMemoryException 内存不足以继续执行

禁用错误注入

# 完全关闭错误/api/loadtest？errorAfter=0# 或将误差百分比设为0/api/loadtest？errorPercent=0

🔍 负荷测试结果诊断

1. Application Insights 积分

  • 前往 Application Insights >表演
  • 按操作名称进行筛选 GET /api/loadtest
  • 观察响应时间百分位数（第50页，第95页，第99页）
  • 检查可配置的随机错误阈值（默认：120秒）的失败率

2. Azure 监测指标

  • 请求 - 总请求数和速率
  • 响应时间 - 平均和百分位延迟
  • HTTP 服务器错误 - 随机异常导致的5xx错误
  • CPU 百分比 - 应随负载增加
  • 记忆工作集 - 应随bufferSizeKb增加

3. 负载测试统计端点

测试期间查询统计端点：

GET /api/loadtest/stats { “currentConcurrentRequests”：127， “处理请求总数”：45230， “totalExceptionsThrown”： 23， “平均响应时间Ms”：847.5 }

📤 响应格式

每次成功请求返回：

{ “elapsedMs”：1247， “concurrentRequestsAtStart”： 42， “降级延迟应用Ms”：8500， “工作迭代完成”：200， “memoryAllocatedBytes”：20480000， “工作完成”：真， “消息”：“负载测试完成时间为1247毫秒” }

响应场

场地 描述
elapsedMs 总请求时长（毫秒）
concurrentRequestsAtStart 请求开始时并发请求数量
degradationDelayAppliedMs 因超额而追加延迟
workIterationsCompleted 完成的SHA256哈希迭代次数
memoryAllocatedBytes 为本次请求分配的内存缓冲区大小
workCompleted 所有工作是否成功完成
message 人类可读的完成信息

✅ 最佳实践

DO：

  • ✅ 从默认参数开始建立基线
  • ✅ 在负载测试中监控 Azure 指标
  • ✅ 用 /api/loadtest/stats 端点来跟踪并发
  • ✅ 逐渐增加负载以找到断裂点
  • ✅ 将载荷测试结果与Application Insights相关
  • ✅ 用真实的负载模式测试（加量、稳态、减速）

不要这样做：

  • ❌ 从极端参数开始
  • ❌ 使用大缓冲区大小Kb时忽略内存度量
  • ❌ 如果你想看到真实的失败，可以运行 errorAfter=0 的扩展测试
  • ❌ 假设一个 SKU 的结果适用于另一个
